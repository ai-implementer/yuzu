# 品質チェック

コードにフォーマッタと linter があるように、yuzu は設計書にも同じ道具を
用意しています。3 コマンドとも終了コードは **0 = 違反なし / 1 = 違反あり /
2 = 実行エラー**の規約で、そのまま CI に組み込めます。

## yuzu fmt — 決定的整形

```bash
yuzu fmt          # 正規形へ整形して書き込む
yuzu fmt --check  # 差分があれば終了コード 1（CI 用）
```

AST 経由の決定的な整形で、見出しの ATX 化・箇条書きの `-` 統一などを
行います。次の不変条件を守ります:

- **frontmatter は生テキストをバイト温存**（整形対象は本文だけ）
- **冪等**（2 回かけても結果は同じ）
- 差分のないファイルには書き込まない（mtime 温存）

## yuzu lint — 文書規約と表記ゆれ

```bash
yuzu lint        # 違反を行番号付きで報告
yuzu lint --fix  # 表記ゆれ系の変換候補を自動適用
```

文書規約（h1 の重複・見出しレベルの飛び・frontmatter の未知キー・
コードブロックの表示メタ検査 = `showLineNumbers` のタイポや範囲外の
行ハイライトの警告）に加えて、日本語の表記ゆれを検出します。

### 用語統一の辞書（lint.terms）

「正しい表記 → ゆれ表記」の辞書を書くと、本文・見出しのゆれを
行番号・列番号付きで報告します（コードブロック・URL は対象外）:

```jsonc
"lint": { "terms": { "サーバ": ["サーバー"], "ユーザ": ["ユーザー"] } }
```

この辞書は[検索のクエリ拡張](search.md)にも使われます。

### 組み込みルール（lint.rules）

辞書がなくても、機械的なゆれは既定で検出します（ルール単位で無効化可能）:

| ルール | 検出対象 | 例 |
| --- | --- | --- |
| `fullwidth-alphanumeric` | 全角英数字 | `Ｗｅｂ１２３` → `Web123` を提案 |
| `halfwidth-kana` | 半角カナ | `ﾃﾞｰﾀ` → `データ` を提案 |
| `katakana-choon` | 長音符ゆれの混在 | プロジェクト内の多数派へ寄せる警告 |

`--fix` は変換候補をまとめて適用します（コードブロックは触らず、冪等で、
差分のないファイルには書き込みません。長音符ゆれは多数派へ統一し、
同数のときは正解を決められないため報告のみです）。

### ディレクトリ階層の制限（lint.maxDirectoryDepth）

`content` 配下のディレクトリの深さを制限できます（直下 = 0）。サイトの
構成が意図せず深くなるのを防ぎます。

## yuzu check — 統合チェック

```bash
yuzu check
```

lint のすべてに加えて、**内部リンク・アンカー切れ**（ページ間リンク・
`#見出し` 参照・画像などの添付ファイル参照）を行番号付きで報告する
統合コマンドです。CI ではこれ 1 つを回せば十分です。

> [!NOTE]
> fmt / lint / check は `draft: true` の下書きページも対象です
> （公開前の文書も同じ規約で保てます）。
> 
> このサイト自身も、デプロイ前に `yuzu check` を通しています
> （[配信とデプロイ](deploy.md)参照）。
