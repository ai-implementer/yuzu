# インクリメンタルビルドの内部設計

`yuzu build` / `yuzu dev` は常時インクリメンタルです（`.yuzu/cache/` に
ページ単位キャッシュ）。設計の要点は「**何をキャッシュしないか**」に
あります。

## キャッシュ対象は高価なページ派生物だけ

キャッシュするのはページ単位の派生物（メタ・本文 HTML・検索 tf・llms 用の
正規化 Markdown）だけです。ナビ・検索インデックスの集約（fst / シャード）・
llms.txt の連結・テンプレート合成は**毎回全実行**します。

クロスページ依存（あるページの変更が他ページの出力へ波及する経路）は
集約とテンプレート段に閉じているため、この分離だけで**依存解析なしに
正しさを保てます**。集約は全ページ分を毎回やり直しても軽い、という
実測に基づく割り切りです。

## 無効化キーは 3 層

| 層 | キー | 無効化の範囲 |
| --- | --- | --- |
| 1 | **envKey**（設定・yuzu バージョン・トークナイザモデルの内容ハッシュ） | 不一致で全破棄 |
| 2 | **routesKey**（rel → route の集合） | ページ増減・改名時に本文 HTML だけ全破棄（`.md` リンク解決の入力のため） |
| 3 | **sourceHash**（ページ単体の内容） | 変更されたページだけ再計算 |

不整合・破損・`.yuzu/cache/` の削除は、常に「キャッシュなし ＝ フルビルド」へ
縮退します。キャッシュを消すことはいつでも安全です。

## 出力は compare-before-write

出力ファイルは内容が一致していれば書き込みません（mtime 温存。
`yuzu fmt` と同じ思想です）。書き出した dist 相対パスは
`output-manifest.json` に記録し、前回との差分で**削除ページの孤児出力を
自動掃除**します。

## ページ並列化と決定性

render（本文 HTML 生成〜テンプレート〜書き出し）と index（ページごとの
トークナイズ）のページループは rayon で並列化しています。集約は
ページ順の直列のままです。

- ハイライタのページ内状態はページローカルな `PageCodeRenderer` に分離し、
  `Cell` の `!Sync` が誤共有をコンパイル時に防ぎます
- インクリメンタル結果はフルビルドと**バイト同一**、並列化の出力は
  スレッド数に依らず**バイト同一**（決定性ゲートで検証済み）です

実測（release ビルド）では、ハイライトと SSR 図が支配的なコーパス
（201 ページ）でフルビルドが約 3 倍速くなり、テキスト主体のコーパスでも
トークナイズの並列化が効きます。

## ライブラリとしての形

`RenderCtx` / `IndexCtx` の**全フィールド None ＝ 従来のフルビルドと
同一動作**という契約にしており、キャッシュの配線は cli 層
（`BuildSession`）だけの責務です。ライブラリ単体テストはキャッシュなしの
形で書け、キャッシュ機構がライブラリの API を汚しません。
